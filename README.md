# Дашборд аналитики оттока клиентов (Churn Analytics)

**Публичный проект** для генерации синтетических данных, расчёта метрик оттока и визуализации ключевых показателей.  
Включает PostgreSQL (100 000+ пользователей), 13 интерактивных графиков Plotly и веб‑дашборд.  
Всё запускается одной командой через Docker — данные и графики генерируются автоматически.

## 🚀 Быстрый старт

```bash
git clone https://github.com/DmitriyKabanov-AI/churn_analytics_full.git
cd churn_analytics_full
docker-compose up --build
```

После завершения сборки:

Grafana → http://localhost:3001 (логин/пароль admin/admin)

Контейнер python сам:

сгенерирует тестовые данные (при первом запуске),

построит все 13 графиков Plotly,

соберёт единый дашборд output/index.html.

Контейнер web (nginx) отдаёт статический дашборд.
Повторные запуски не пересоздают данные (если нужно с нуля — docker-compose down -v).

## 📈 Часть 1. Графики Plotly (13 визуализаций)

| № | График | Что показывает | Как анализировать | 💡 Бизнес-вывод |
|---|---------|---------------|-------------------|-----------------|
| 1 | **Income vs Churn** (Box plot) | Распределение дохода у оставшихся (Retained) и ушедших (Churned) | Сравнить медианы и разброс: Retained ≈55k, Churned ≈38k | Клиенты с низким доходом уходят чаще → предложить скидки или продление триала |
| 2 | **Plan vs Churn** (Stacked bar) | Число клиентов по тарифам + доля оттока в каждом | Искать тарифы с наибольшим % оттока (а не абсолютным числом) | В премиум-планах % оттока может быть выше → проверить цену/функционал |
| 3 | **Distribution of Income** (Histogram + KDE) | Плотность распределения дохода всех пользователей | Пик ~60k, лёгкий левый сдвиг | Большинство клиентов зарабатывают 50–70k → ориентир для ценообразования |
| 4 | **User Activity** (log scale) | Распределение событий на пользователя (логарифмическая шкала) | Большинство <10 событий; длинный хвост сверхактивных | Повышать вовлечённость: пуши, онбординг, бонусы за действия |
| 5 | **Correlation Matrix** | Корреляции: income, events, purchases | events↔purchases >0.8; income слабо связан | Стимулировать любые действия пользователя → рост покупок и выручки |
| 6 | **Events vs Churn** (Box plot) | Количество событий: у оставшихся и ушедших | Медиана у Retained 10–12 событий, у Churned 5–6 | Низкая активность = сигнал оттока → авто-детект и возврат «затихающих» |
| 7 | **ROC‑Curve** | Качество моделей (TPR vs FPR) | LR и RF: AUC≈0.5 (не работают); GB/XGB: целевые AUC>0.82 | На синтетике отток непредсказуем; в реале нужны качественные фичи |
| 8 | **Feature Importance** | Важность признаков в Random Forest | Топ: income, events, purchases; регионы не важны | При сборе данных приоритет — поведенческие признаки, а не демография |
| 9 | **Confusion Matrix** | TP/TN/FP/FN для Random Forest | Recall=9% (пропущено 330 из 3646 ушедших) | Оптимизировать под recall — лучше ложная тревога, чем пропустить отток |
| 10 | **Model Metrics Table** | AUC/Accuracy/Precision/Recall/F1 для 4 моделей | LR/RF провалились (AUC~0.5); GB/XGB — эталон (AUC>0.82) | В продакшене использовать градиентный бустинг на реальных данных |
| 11 | **Cumulative LTV** | LTV по когортам регистрации (по месяцам) | Старые когорты (2025-05) имеют бόльший LTV, чем новые | Качество привлечения падает → аудит маркетинговых каналов и онбординга |
| 12 | **Rolling ARPU** | Ежедневный ARPU + 7-дневное скользящее среднее | Нисходящий тренд, подтверждённый SMA | Монетизация снижается → исследовать частоту покупок и средний чек |
| 13 | **Churn Dynamics** | Ежедневное количество завершённых подписок | Волатильность: пики до 400/день, в среднем 100–200 | Пики могут совпадать с окончанием триала → добавить напоминание за 3 дня |

|--------|---------------|-------------------|-----------------|

| **ARPU** (Avg Revenue Per User) | Средняя выручка на пользователя в день | Тренд на снижение, подтверждён скользящим средним | Тревожный сигнал по монетизации → проверить конверсию и средний чек |
| **Conversion to Purchase (%)** | % пользователей, совершивших покупку | 19–22% (нереалистично высоко для реальных данных) | Артефакт синтетики; в продакшене считать конверсию по полной воронке |
| **Retention Heatmap** | Удержание по когортам (цветом показан rate) | Месяц 1: >90% (аномалия); Месяц 2–3: 60–64% | Реальное удержание ~60% → улучшать онбординг и ценность продукта на старте |
| **RFM-сегменты** | Распределение пользователей по кодам RFM (111–444) | Чаще всего: 111 (низкая ценность), 444 (высокая ценность) | Поощрять сегмент 444; возвращать сегмент 111 персональными акциями |
| **LTV by Cohort** | Кумулятивный LTV по месяцу привлечения | Когорта 2025-05: ~35; новые: ~20 | Качество новых пользователей упало → усилить маркетинг и продукт для новичков |

## 🧠 Итоговое резюме для бизнеса

### 🔑 Ключевые метрики в одном взгляде

| Метрика | Значение | Статус | Рекомендация |
|---------|----------|--------|--------------|
| **Ежемесячный отток** | ~3.55% | 🟡 Умеренный | Внедрить предиктивную аналитику для раннего предупреждения |
| **Годовой отток** | ~30% | 🔴 Высокий | Приоритет: удержание > привлечение |
| **Удержание (Месяц 2)** | ~60% | 🟡 Есть потенциал | Усилить онбординг и ценность продукта на ранних этапах |
| **События ↔ Покупки** | Corr > 0.8 | 🟢 Сильная связь | Геймификация, награды за действия → рост выручки |
| **Тренд ARPU / LTV** | Снижение | 🔴 Тревожно | Аудит новых когорт, оптимизация монетизации |
| **AUC моделей (синтетика)** | ~0.5 | ⚪ Неинформативно | Для реальных данных: XGBoost + поведенческие фичи |

### 🎯 Топ-5 рекомендаций к действию

1. **Сегментировать причины оттока** — опросить ушедших пользователей, чтобы понять *почему* они ушли (цена, функционал, поддержка).
2. **Запустить триггерные рассылки** — автоматически таргетировать пользователей с <5 событий/неделю персональными офферами.
3. **A/B-тестировать онбординг** — добавить интерактивный тур и быстрый «вау-эффект» в первые 24 часа для роста раннего удержания.
4. **Внедрить гибкие тарифы** — протестировать нижние тарифные планы или pay-as-you-go для сегментов с низким LTV.
5. **Подготовить ML-пайплайн для продакшена** — собирать поведенческие события, инженерить признаки, обучать модели градиентного бустинга на реальных данных.

### 📊 SQL-запросы для Grafana

```sql
-- DAU (Daily Active Users)
SELECT date::timestamp as time, dau as value 
FROM daily_metrics 
ORDER BY date;

-- ARPU (Average Revenue Per User)
SELECT date::timestamp as time, arpu as value 
FROM daily_metrics 
ORDER BY date;

-- Conversion to Purchase
SELECT date::timestamp as time, conversion_to_purchase as value 
FROM daily_metrics 
ORDER BY date;

-- Retention Heatmap
SELECT cohort_month::date as time, months_diff as metric, retention_rate as value 
FROM retention_cohorts 
ORDER BY cohort_month, months_diff;

-- RFM-сегменты
SELECT rfm_score, COUNT(*) as users 
FROM rfm 
GROUP BY rfm_score 
ORDER BY users DESC;

-- LTV по когортам
SELECT purchase_month::date as time, cohort_month, ltv 
FROM ltv 
ORDER BY purchase_month, cohort_month;
```

Пояснение ключевых элементов

analytics/db_utils.py – подключается к БД через переменные окружения (DB_HOST, DB_PORT...). Внутри Docker подставляются значения из docker-compose.yml, локально – значения по умолчанию (localhost:5435).

scripts/build_dashboard.py – запускает все chartX.py, затем читает сгенерированные JSON, очищает шаблон template/index.html от синтетических графиков и вставляет реальные. Результат – output/index.html.

scripts/run_pipeline.py – точка входа для контейнера python: ждёт БД, при отсутствии данных вызывает generate_churn_data, затем запускает build_dashboard.py.

output/ – монтируется как том, поэтому сгенерированные файлы остаются после остановки контейнера. Веб‑сервер (nginx) читает файлы именно отсюда.

template/index.html – содержит стили, боковое меню, навигацию и пустые div для графиков. В процессе сборки из него удаляются синтетические Plotly.newPlot и вставляются реальные.
